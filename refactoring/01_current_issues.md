# Current Issues

## 1. 엔트리포인트 집중

현재 [src/main.py](/D:/vscode/safetycompanion/src/main.py)는 아래 책임을 동시에 맡고 있다.

- CLI parser 정의
- config 로딩과 실행 컨텍스트 생성
- manifest 상태 기록
- ingest 실행
- index build 실행
- query 실행
- eval 실행

이 구조는 작은 프로젝트에서는 빠르지만, 단계가 늘어나면서 다음 문제가 발생한다.

- 변경 영향 범위를 예측하기 어렵다.
- 특정 단계만 읽고 수정하고 싶어도 `main.py` 전체를 열어야 한다.
- 테스트 대상이 도메인 로직이 아니라 진입점 파일에 섞여 보인다.

## 2. 경로 정의 분산

프로젝트의 데이터 경로는 `config`에 정의되어 있지만, 인덱스 관련 경로와 일부 정책 파일 경로는 여러 모듈에서 직접 조합하고 있다.

대표적으로 다음 문제가 있다.

- `RunContext`는 `output_root`와 config 경로를 다루지만 인덱스 경로는 모르고 있다.
- `QueryService`는 `indexes/*` 경로를 직접 알고 있다.
- 정책 파일 경로는 config에 있으면서도 별도 함수에서 직접 조합한다.

이 상태에서는 경로 정책을 바꾸려 할 때 수정 지점을 놓치기 쉽다.

## 3. 애플리케이션 계층 부재

현재 구조는 `common`, `parse`, `ingest`, `retrieval`, `qa`, `eval` 같은 도메인 계층은 있지만, 이를 엮는 명시적 애플리케이션 계층이 약하다.

결과적으로 CLI 파일이 사실상 워크플로 계층 역할까지 맡고 있다.

필요한 구조는 아래와 같다.

- `main.py`: 실행 진입점
- `cli`: 인자 해석과 command dispatch
- `workflows`: 여러 도메인 모듈을 조합하는 실행 절차
- `domain modules`: 실제 기능 구현

## 4. 신규 기여자 관점의 진입 난도

신규 기여자가 처음 읽을 때 가장 먼저 부딪히는 질문은 보통 다음과 같다.

- 어디서 시작해야 하나
- 어떤 경로가 config 기반이고 어떤 경로가 하드코딩인가
- query 하나가 어떤 모듈들을 지나가는가

이 질문에 현재 구조는 즉답하기 어렵다. 리팩터링의 목표는 단순히 파일을 나누는 것이 아니라, 이 질문들에 파일 구조만 보고도 답할 수 있게 만드는 것이다.
